AI Agent Pipeline — LangGraph StateGraph
Agents AGT-01 through AGT-06: Full Implementation Specifications
All 6 agents are Python async functions registered as nodes in a single LangGraph StateGraph, running within one FastAPI application. AGENT-C handles both chatbot management and department information retrieval. AGENT-P applies internal content filters before every retrieval call. Emergency detection (AGENT-E) is a LangGraph interrupt node with highest priority.

Purpose & Role
AGENT-R is the single, mandatory entry point for every query entering the AI pipeline — regardless of platform, stream, or user role. It performs four sequential operations: emergency keyword detection, language classification, intent classification, and downstream routing. It generates no content.
Hard constraint: Emergency detection runs first, before any other step. If any emergency keyword matches, is_emergency=True is set and AGENT-E is called immediately. No LLM call is made for emergency queries.
Classification Pipeline
1. Emergency CheckRegex against EN/FR keyword set. If match → AGENT-E immediately, stop.
→
2. Language Detectlangdetect. EN/FR. Confidence threshold 0.70; fallback EN.
→
3. Intent ClassifyGPT-4o structured output: navigation | information | procedure | unknown.
→
4. Context BuildAssemble AIHPSState: intent, language, platform, stream, user_role from JWT.
Routing Table
emergency→ AGENT-EHighest priority. Pure regex — no LLM. All other agents bypassed.
navigation→ AGENT-NRoute: AGENT-R → AGENT-N → AGENT-O. Deterministic DB retrieval.
information→ AGENT-C (DB)Department info. AGENT-C queries departments table directly — no LLM, no knowledge base. Route: AGENT-R → AGENT-C → AGENT-O.
procedure (chatbot)→ AGENT-CAGENT-C reformulates → AGENT-P (internal filters + RAG) → AGENT-O.
procedure (search/browse)→ AGENT-PAGENT-P applies internal filters then executes retrieval. Route: AGENT-R → AGENT-P → AGENT-O.
unknown→ AGENT-OClarification prompt in detected language. No retrieval.

Purpose & Role — Two Responsibilities
1. Chatbot management — activated when chatbot_mode=True. Maintains multi-turn conversation context, reformulates ambiguous queries into precise retrieval requests, and presents AGENT-P's structured procedure responses in natural conversational format. Does not retrieve procedures — delegates to AGENT-P (which applies internal filters).
2. Department information retrieval — activated when AGENT-R classifies intent as 'information'. Retrieves department metadata (services, hours, contacts, location) directly from the PostgreSQL departments table. No LLM call, no knowledge base query — pure deterministic DB retrieval.
Chatbot Session Design
  • Session key: session:{user_id}:{platform} in Redis. TTL: 30 minutes, refreshed on every message.
  • History: Array of {role, content}. Max 10 turns retained (sliding window).
  • Persistence: Survives app close/reopen within TTL. Fresh start on expiry.
  • Clarification gate: If reformulated query confidence < 0.60 → return one clarifying question, do not call AGENT-P.
Department Info Retrieval (replaces AGENT-I)
  • Direct SELECT from departments table using semantic name matching (sentence-transformers cosine similarity ≥ 0.65).
  • Returns: name, services, operating hours, location, contact details — in detected language.
  • No match (score < 0.65): returns standard redirect message. No LLM generation.
  • The in-memory department name embedding index is shared with AGENT-N — loaded once at startup.

Purpose & Role
AGENT-N provides hospital wayfinding — directions from any entry point to any mapped department. Fully deterministic: all directions are authored by administrators, stored in PostgreSQL, and retrieved verbatim. Zero hallucination risk. Semantic destination resolution uses a pre-computed in-memory embedding index shared with AGENT-C.
Destination Resolution Algorithm
  • Startup: Encode all department names and informal_names[] with paraphrase-multilingual-MiniLM-L12-v2. Cache in memory. Shared with AGENT-C.
  • Resolution: Encode destination phrase from query. Cosine similarity against in-memory index. Match threshold ≥ 0.65.
  • No match: "Department not found. Please ask at the main reception desk." — no guessing below threshold.
  • Language: Retrieve path matching detected language. Fallback to alternate with notice.
Phase 1 Mapped Destinations (minimum 10)
  • Main Entrance → Emergency Department
  • Main Entrance → Blood Bank
  • Main Entrance → ICU
  • Main Entrance → Surgery / Operating Theatre
  • Main Entrance → Maternity Ward
  • Main Entrance → Infection Control
  • Reception → Laboratory · Pharmacy · Outpatient Registration · Radiology
  • Emergency → Blood Bank (internal) · Emergency → ICU (internal transfer)

Purpose & Role
AGENT-P is the RAG/retrieval engine and internal content governance layer for all procedure queries across both streams. It replaces both the former AGENT-F (content filtering) and the former AGENT-P (pure retrieval). Before executing any retrieval, AGENT-P builds and applies mandatory content filters from the routing state. After filtering, it executes parallel semantic and full-text search and generates a structured, grounded response using GPT-4o.
Clinical safety constraint: If the maximum similarity score across all retrieved chunks is below 0.40, AGENT-P must return the standard no-result message and set had_result=False. No answer shall be generated from general LLM knowledge. This is a hard architectural constraint.
Internal Filtering (replaces AGENT-F)
Before any knowledge base query, AGENT-P builds a filter object from the routing context:
  • stream_target filter: Only entries where stream_target = current_stream OR 'both'.
  • Role filter (Stream B only): applicable_roles must contain the JWT user_role. Stream A has no role filter — content is controlled by stream_target only (no authentication available).
  • Status filter: Only status = 'published'. Draft, pending, and archived entries are always excluded.
  • Language preference: Entries in the detected language ranked higher; alternate language supplements when < 3 results in preferred language.
Combined Retrieval Architecture
Apply FiltersBuild filter dict from state. Stream A: stream_target + status. Stream B: + applicable_roles from JWT.
→
Semantic (FAISS)text-embedding-3-small query encoding. ANN search with filters. Top-20 with cosine scores.
+
FTS (PostgreSQL)ts_rank against search_vector TSVECTOR with filters. Normalised rank scores.
→
Weighted Mergecombined = 0.7×semantic + 0.3×fts. Sort desc. Top-5 passed to generation.
Structured Response Format (Stream B)
  • 1. AI Disclaimer — non-suppressible, rendered first by AGENT-O
  • 2. Procedure Summary — from retrieved content
  • 3. Step-by-Step Actions — from steps JSONB, numbered
  • 4. Required Approvals — from approval_required flags per step
  • 5. Compliance / Legal Notes — from compliance_annotations JSONB
  • 6. Risk Level — sets risk_flag if High/Critical for AGENT-O colour badge
  • 7. Escalation Pathway — from linked escalation_pathways if present
  • 8. Source Citations — procedure title + version; tappable links
Workflow Mode
Activated from the procedure detail screen. AGENT-P restructures the retrieved procedure into sequential step-by-step execution. Each call advances one step, surfacing the step-specific compliance note and approval gate. Step state stored in Redis per user session.

Purpose & Role — Critical Safety Component
AGENT-E handles all emergency queries with the highest system priority. When AGENT-R sets is_emergency=True, the graph routes immediately to AGENT-E — bypassing AGENT-C, AGENT-N, and AGENT-P entirely. Emergency content is pre-composed and stored in PostgreSQL — zero LLM calls on the emergency path, eliminating both generation latency and hallucination risk.
Hard constraints (SRS FR-E-05, FR-E-06): (1) Emergency responses are NEVER compressed by AGENT-O on any platform. USSD emergency responses use as many screens as needed. (2) Every emergency activation is immediately logged to SVC-06 with escalation_triggered=True. (3) Delivered within 3 seconds of message receipt on all platforms.
Stream A Emergency Response
  • Pre-composed per language (EN/FR). Retrieved from emergency_content table.
  • Contains: (1) immediate safety actions, (2) landmark-based directions to Emergency Dept entrance, (3) hospital emergency contact number.
  • WhatsApp: full content with 🚨 and bold. SMS: ~200 chars (emergency exempt from 160-char limit). USSD: END response — user must act, not navigate menus.
Stream B Emergency Response
  • Emergency SOP retrieved by direct pre-defined query — bypasses all role/stream filters (all staff have emergency access).
  • Returns: emergency SOP steps, required approvals, complete escalation chain.
  • Pre-defined query eliminates full RAG pipeline latency to meet the 3-second SLA.
