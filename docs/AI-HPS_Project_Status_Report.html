1. What Is AI-HPS? (Simple Explanation)

For non-technical readers: AI-HPS is a smart assistant system for Hôpital Général de Douala. It works on WhatsApp, SMS, a mobile app, and a web portal. A patient can send a message asking "How do I register for surgery?" and the system replies with the correct steps in English or French — instantly. Hospital staff (doctors, nurses, admins) use the mobile app to look up medical procedures, approve new content, and track how the system is being used.

Two Types of Users

Stream A — Patients / Public

Ordinary people contact the hospital via WhatsApp, SMS, or USSD (short code on any phone). No registration or login needed. The AI answers in English or French. If there is an emergency, it responds in under 3 seconds.

Stream B — Hospital Staff

Doctors, nurses, pharmacists, and administrators log in via the mobile app or web portal. They can browse all procedures, get detailed clinical guidance, submit new procedures for approval, and view system analytics.

2. How the System Is Built (Technical Architecture)

For non-technical readers: Think of the system as a building with several rooms, each doing a specific job. There is a security room (login), a knowledge room (procedures), an AI room (the smart brain), a record room (audit logs), and a reception room (the app the user sees). All rooms talk to each other through secure channels.

3. What the System Can Already Do

For non-technical readers: The following features are coded and ready. They have not been fully tested together yet, but all the pieces are built.

AI & Intelligence

  • Detect medical emergencies in a message and respond in under 3 seconds — without even calling the AI (using pre-cached data)
  • Understand what the user wants: emergency help, navigation directions, general information, or a specific procedure
  • Find the most relevant hospital procedures using both keyword search and semantic (meaning-based) AI search
  • Generate answers grounded only in real hospital documents — it will not make up information
  • Refuse to answer if it cannot find a match above its confidence threshold (0.40 out of 1.0)
  • Detect whether the user is writing in English or French and reply accordingly
  • Format its replies differently depending on the platform: shorter for SMS, full for the web portal

Security & Access Control

  • Secure login with encrypted password and JWT tokens (auto-expire for safety)
  • 10 distinct staff roles: Super Admin, Admin, Doctor, Nurse, Pharmacist, Technician, Radiologist, Receptionist, Intern, Auditor
  • Procedures filtered by role — a nurse sees nurse-relevant content, a doctor sees all clinical content
  • Account lockout after repeated failed login attempts
  • Token blacklist — logging out truly invalidates the session

Procedure Management

  • Create, edit, and version hospital procedures with categorization by department
  • Multi-stage approval workflow — a procedure must be approved before it goes live
  • Risk level tagging: High, Medium, Low
  • Target audience tagging: Stream A (patients) or Stream B (staff) or both
  • Full-text and AI semantic search across all procedures
  • Navigation paths — directions from one hospital location to a department
  • Emergency content entries — pre-written responses for critical situations

Mobile App (Hospital Staff)

  • Login screen with credential validation
  • Home screen with quick access chips and recent procedures
  • AI chat interface (assistant) with structured response cards
  • Procedure browser with filters and detail view
  • Search interface with full-text filters
  • Admin panel: manage procedures, run approvals, view compliance log
  • User profile and password change
  • Bilingual interface: English and French
  • Offline indicator banner (app shows warning when internet is lost)

Database & Infrastructure

  • Complete PostgreSQL database with 17 tables across 5 organized schemas
  • Redis fast-memory cache (emergency content is pre-loaded here)
  • RabbitMQ message queue for processing audit events in the background
  • Nginx API gateway with rate limiting and security checks
  • Docker setup for all infrastructure components
  • WHO medical PDF documents stored per department (5 departments, 15 PDFs)

4. What Is Not Yet Done

Important: The missing pieces do not mean the system is broken — the foundation is solid. These are the final steps that connect the system together, load it with real data, and deliver it to users on all platforms.

Critical Missing Items

  • PDF Ingestion Not Run: The 15 WHO medical PDFs are stored but have not yet been loaded into the database and AI search index. The AI cannot answer procedure questions from these documents yet.
  • SVC-04 (Notifications) Not Built: Push notifications to the mobile app are not implemented. Users will not receive alerts for approvals, updates, or messages.
  • SVC-05 (Analytics) Not Built: The analytics dashboard backend is missing. The app has the admin analytics screen, but it has no real data to display.
  • WhatsApp/SMS/USSD Gateway Not Connected: The AI is ready to format replies for these channels, but the actual integration with a telecom or WhatsApp Business API gateway has not been set up.
  • Frontend Not Connected to Backend: The mobile app currently runs on mock (fake) data. It has not been wired up to the real backend API services.
  • Full Stack Never Run Together: All services have been built individually but have not been launched and tested as one unified running system.

Secondary Missing Items

  • Web portal for patients and public access (the mobile app exists, but a browser-based portal is not built)
  • Offline sync — the mobile app shows an offline warning but cannot yet sync data when the connection comes back
  • End-to-end integration tests (automated tests that simulate real users)
  • Deployment to a real server (currently only runs on a developer laptop)
  • User acceptance testing with real HGD staff
  • Security audit and penetration testing before going live

5. Roadmap — Steps to Complete the Project

Reading guide: Steps are ordered by priority. Red numbers = must be done first (blocking everything else). Orange numbers = important but can be done in parallel. Green numbers = final polish before delivery.
1

Load Hospital Documents into the AI (PDF Ingestion)

The 15 WHO medical PDF documents are stored in the PROCEDURES/ folder but have not been loaded into the database or the AI search engine yet. This is the single most important missing step — without it, the AI has no medical knowledge to draw from.

  • Run the ingestion script that extracts text from all PDFs
  • Store the text in the PostgreSQL procedures table
  • Build the vector (AI) search index from the text
  • Verify the AI can find and answer questions using the documents
Technical: Run backend\venv\Scripts\python scripts\ingest_procedures.py from the project root. Then start SVC-07 and AGENT-P and run a test query. Requires OpenAI API key to be set in .env.
2

Start All Services and Run the Full Stack

Each service (Auth, Procedures, Audit, KB Sync, Agent Pipeline) needs to be started and confirmed to be talking to each other correctly. This is the first time the complete system will run together.

  • Start Docker (Redis, RabbitMQ, Nginx) via docker-compose
  • Start all backend services (SVC-02 through SVC-07 + Agent Pipeline)
  • Run a login request, then ask the AI a question end-to-end
  • Fix any startup errors that appear (config mismatches, missing env variables)
Technical: Each service runs on its own port (8010–8015 + 8020). Start with uvicorn services.svc02_auth.main:app --port 8010 etc. Check http://localhost:808X/docs for each service's interactive documentation.
3

Connect the Mobile App to the Real Backend

The mobile app is currently showing fake (mock) data. Every screen needs to be connected to the real API so that it displays real procedures, real user accounts, and real AI responses.

  • Update the API base URL in the frontend to point to the running backend
  • Test the login screen with a real staff account
  • Test the procedure browser loading real procedures from the database
  • Test the AI chat assistant returning real answers
  • Test the admin approval workflow from creation to approval
Technical: Edit frontend/api/client.ts — set baseURL to http://<host>/api. Remove mock imports in screens one by one. Use TanStack Query hooks already wired in frontend/hooks/.
4

Build SVC-04 — Push Notifications Service

When a procedure is approved, updated, or when there is an important message, staff should receive a push notification on their mobile device. This service needs to be built, following the same pattern as the other services.

  • Create backend/services/svc04_notifications/ following the same structure as SVC-02/03
  • Implement an endpoint to register device push tokens
  • Implement a send-notification endpoint that delivers to registered devices
  • Connect to Expo Push API (used by the frontend framework)
  • Wire up triggers: approval event → notification sent
Technical: Use httpx to call Expo's push service at https://exp.host/--/api/v2/push/send. The database table aihps_notifications.push_registrations already exists in schema.sql.
5

Build SVC-05 — Analytics Service

The admin dashboard has a KPI screen but no data behind it. This service will aggregate usage statistics: how many queries per day, which departments are most searched, what questions had no answers (content gaps).

  • Create backend/services/svc05_analytics/
  • Build endpoints: daily query counts, top intents, content gap report, weekly summary
  • The analytics database tables already exist — just need the API layer
  • Connect the admin analytics screen in the mobile app to this service
Technical: Tables aihps_analytics.query_events, aihps_analytics.content_gaps, aihps_analytics.weekly_reports are already in the schema. The agent pipeline already writes query events — just need aggregation endpoints.
6

Connect WhatsApp / SMS / USSD Channels

This is the most visible feature for patients. When a patient sends a WhatsApp message to the hospital number, the AI should reply. The AI already knows how to format replies for these channels — what is missing is the connection to the messaging provider.

  • Choose a messaging gateway provider (e.g., Africa's Talking for USSD/SMS, Twilio for WhatsApp)
  • Create a webhook endpoint that receives incoming messages from the gateway
  • Pass the message to the Agent Pipeline, then send the reply back via the gateway API
  • Test with a real SIM card (USSD) and a real WhatsApp Business account
Technical: Add a /webhook/whatsapp and /webhook/ussd route. For SMS/USSD in Cameroon, Africa's Talking has local operator coverage. For WhatsApp, use the official WhatsApp Business Cloud API (Meta). The Agent Pipeline's AGENT-O already handles platform-specific formatting.
7

Seed the Database with Real Hospital Data

Beyond the WHO PDF documents, the system needs real HGD-specific data: actual department details, operating hours, contact numbers, navigation directions, and at least one staff admin account to log in with.

  • Create a seed script that inserts HGD departments, categories, and initial procedures
  • Add navigation paths for the hospital layout (directions between departments)
  • Create the initial super-admin account for system setup
  • Review and adapt the WHO procedures to match HGD's actual practices
Technical: Write backend/scripts/seed_hgd_data.py. Use SQLAlchemy models in backend/shared/models/ to insert records. Start with departments, then categories, then procedure entries.
8

Integration Testing

Once all services are running and connected, the full system needs to be tested end-to-end to catch bugs that only appear when all parts work together.

  • Test a patient sending an emergency message via the simulated WhatsApp channel — does it respond in under 3 seconds?
  • Test a staff member logging in, searching for a procedure, and reading the full steps
  • Test an admin creating a procedure and sending it through the approval workflow
  • Test that audit logs record every action correctly
  • Test that the AI refuses to answer questions it cannot find evidence for
  • Test in French: are all responses correct in French?
Technical: Write test scripts using pytest and httpx. Consider writing a Postman collection for manual API testing. Test each agent in isolation first, then the full pipeline.
9

Deployment to a Real Server

Currently the system only runs on a developer's Windows laptop. It needs to be moved to a server that is always online, accessible from the internet, and able to handle real hospital traffic.

  • Choose a hosting environment (a cloud server or a dedicated hospital server)
  • Configure the server with the same Docker setup
  • Set up a real domain name and HTTPS (SSL certificate for security)
  • Set up automatic backups of the database
  • Set up monitoring alerts so the team knows if the system goes down
Technical: The Docker Compose file is already written. For cloud deployment, a VPS running Ubuntu with Docker Engine will work. Recommended: DigitalOcean Droplet or an on-premise server at HGD. Use Certbot (Let's Encrypt) for HTTPS. Minimum spec: 4 GB RAM, 2 vCPUs, 40 GB SSD.
10

User Acceptance Testing at HGD

Before final submission, the system should be tested by real staff and patients at Hôpital Général de Douala to confirm it meets their needs and is easy to use.

  • Run a pilot with a small group of nurses and receptionists
  • Observe and record any usability issues
  • Collect feedback on missing procedures or unclear answers
  • Fix reported issues and add missing content
  • Document the test results for the academic thesis
Technical: Prepare a UAT feedback form. Monitor the system logs and analytics during the pilot to identify content gaps automatically. This maps to the SRS acceptance criteria in context/AI_HPS_SRS_v3.1.docx.

7. Technologies Used

Backend Language

Python 3.11 — FastAPI framework for building the API services. Chosen for speed, simplicity, and excellent AI library support.

AI Framework

LangGraph + OpenAI GPT-4o-mini — LangGraph orchestrates the 6 agents in a pipeline. GPT-4o-mini generates answers from retrieved procedures.

Database

PostgreSQL 15 — Stores all users, procedures, audit events, analytics. Includes built-in full-text search.

Cache

Redis 7.2 — Fast in-memory storage for emergency content (pre-loaded for instant replies) and intent caching.

Message Queue

RabbitMQ 3.12 — Handles background processing of audit log events without slowing down the main responses.

Mobile App

Expo (React Native) 56 — Builds one codebase that works on both Android and iOS. The staff app runs on phones and tablets.

API Gateway

Nginx 1.25 — The single secure entry point. All traffic passes through here for authentication checks and rate limiting.

Vector Search

SentenceTransformers + FAISS — Converts text into mathematical vectors so the AI can find documents by meaning, not just keywords.
